Design Patterns

MVVM and MVI architecture patterns — when to use each and full examples.

MVVM

📖 Docs

Model-View-ViewModel. The ViewModel exposes observable state; the View observes and renders. The Model (Repository) handles data.

Architecture diagram:
View (Composable)
  ↕ observes StateFlow
ViewModel
  ↕ calls suspend functions / collects Flow
Repository
  ↕
Remote (Retrofit) + Local (Room)
Kotlin — MVVM
📚 Deep Dive

Key MVVM rules:

  • View knows about ViewModel, not vice versa.
  • ViewModel knows about Repository, not View or Context.
  • Repository is the single source of truth — decides whether to fetch remote or return cached data.

Multiple UI state variables vs single state class: For complex screens, prefer a single data class UiState(...) with _state.update { it.copy(...) }. Simpler screens can use individual StateFlows.

ViewModels and Context: Never pass an Activity context to a ViewModel — it will outlive the Activity. Use @ApplicationContext via Hilt if you need context.

MVI

Model-View-Intent. All user actions are sealed class Intents. A single immutable State is the only source of truth. State transitions are predictable and testable.

Architecture diagram:
View (Composable)
  → dispatches Intent (user actions)
ViewModel.processIntent()
  → produces new State (immutable copy)
  → (optionally) emits one-off Effect (navigate, snackbar)
View
  ← renders State
Kotlin — MVI
📚 Deep Dive

Intent (Action) naming: Name intents as user actions: LoadUsers, SearchQuery(query), DeleteUser(id), RetryClicked.

Effect / SideEffect: One-off events that shouldn't be replayed on config change (navigate, show toast). Use a Channel or SharedFlow(replay=0).

State must be immutable: Use data class with .copy(). Never mutate state directly.

Testability: MVI shines here — given an Intent, assert the output State. No need to mock UI interactions.

MVVM vs MVI

AspectMVVMMVI
State managementMultiple StateFlows or one state classSingle immutable state class
User actionsDirect function calls on ViewModelSealed Intent class dispatched to ViewModel
ComplexityLower — simpler to get startedHigher upfront — but scales well
TestabilityGood — mock repositoriesExcellent — pure state machine
One-off eventsSharedFlow or ChannelEffect / SideEffect channel
DebuggingStandardEasy — log every Intent → State transition
Best forMost apps, simple to medium complexityComplex screens, team projects, high testability
Interview tip: Both patterns use a ViewModel and Repository. The key difference is how state is managed and how the View communicates with the ViewModel. In MVVM, the View calls functions directly (vm.loadUsers()). In MVI, the View sends typed intents (vm.dispatch(Intent.LoadUsers)).
