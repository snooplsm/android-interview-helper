Scopes

📖 Docs

Every coroutine runs inside a CoroutineScope that defines its lifetime. Android provides lifecycle-aware scopes.

Kotlin
📚 Deep Dive

viewModelScope: Cancelled in ViewModel.onCleared() — ideal for data loading.

lifecycleScope: Cancelled when the Activity/Fragment is destroyed. Use repeatOnLifecycle for Flow collection so it stops when the app is backgrounded.

rememberCoroutineScope (Compose): Scoped to the composable's lifecycle. Use for user-triggered one-off events (scrolling, snackbars).

GlobalScope — avoid: Not scoped to any lifecycle. Leaked coroutines are a common bug.

launch & async

📖 Docs

launch is fire-and-forget (returns Job). async computes a result (returns Deferred, get result with .await()).

Kotlin

withContext & Dispatchers

📖 Docs

Switch the coroutine's thread pool without launching a new coroutine. withContext suspends the current coroutine and resumes on the original dispatcher when done.

Kotlin
📚 Deep Dive

Dispatchers.Main: Android main thread. Use for UI updates.

Dispatchers.IO: Optimized for blocking I/O (network, disk). Elastic thread pool (up to 64 threads).

Dispatchers.Default: CPU-bound work (sorting, parsing). Thread count = CPU core count.

Dispatchers.Unconfined: Starts in the caller's thread, resumes in whatever thread the last suspend function resumed on. Avoid in production.

Best practice in Repository: Always call withContext(Dispatchers.IO) inside repo functions so callers don't need to know about threading.

Cancellation & Timeout

📖 Docs

Cancellation is cooperative — coroutines must check for cancellation at suspension points or explicitly with ensureActive().

Kotlin
Interview tip: Cancellation throws CancellationException — never catch it with a broad catch(e: Exception) without re-throwing. Use catch(e: Exception) { if (e is CancellationException) throw e }.

Error Handling

📖 Docs

Exceptions in launch propagate to the scope and cancel siblings. Use SupervisorJob or supervisorScope to isolate failures.

Kotlin
📚 Deep Dive

CoroutineExceptionHandler: Only catches uncaught exceptions in root coroutines (launched directly in scope). Does NOT catch exceptions from async — those are exposed via .await().

SupervisorJob: Child failures don't cancel siblings or the parent. viewModelScope already uses a SupervisorJob internally.

supervisorScope vs coroutineScope: coroutineScope cancels all children on any failure. supervisorScope lets them fail independently — use when fetching multiple independent resources.
