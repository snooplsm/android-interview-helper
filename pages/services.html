Background work, bound services, foreground services, and WorkManager.

Service

📖 Docs

Runs in the main thread by default — spawn a coroutine or thread for actual background work. START_STICKY restarts the service if killed.

Kotlin
📚 Deep Dive

Return values for onStartCommand:

  • START_STICKY — System restarts service with a null Intent after killing it. Good for music players.
  • START_NOT_STICKY — Not restarted after kill. Good for one-off tasks.
  • START_REDELIVER_INTENT — Restarted with the original Intent. Good for file downloads.

Modern alternative: Prefer WorkManager for deferrable work, ForegroundService for user-visible immediate work.

IntentService (Deprecated API 30)

📖 Docs
Deprecated in API 30. Use CoroutineWorker (WorkManager) instead. Still asked about in interviews.
Kotlin

BoundService

📖 Docs

Allows components to bind, call methods directly, and receive results. Service stops when the last client unbinds.

Kotlin

ForegroundService

📖 Docs

Visible to the user via a persistent notification. Won't be killed by the system. Required for long-running user-initiated tasks (media, navigation, file transfer).

Kotlin

CoroutineWorker / WorkManager

📖 Docs

Best for deferrable, guaranteed background work (sync, uploads, cleanup). Survives process death, device reboots, and respects constraints.

Kotlin

Hilt Worker

📖 Docs

Inject dependencies into WorkManager workers via @HiltWorker + @AssistedInject. Requires a custom HiltWorkerFactory.

Kotlin
