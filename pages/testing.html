TOML / Gradle

Coroutines

📖 Docs

Use runTest to run suspending code in a test. Swap Dispatchers.Main with UnconfinedTestDispatcher or StandardTestDispatcher via Dispatchers.setMain().

Kotlin
📚 Deep Dive

StandardTestDispatcher: Coroutines do NOT run automatically. You advance time manually with advanceUntilIdle(), advanceTimeBy(ms), or runCurrent(). Gives precise control over ordering.

UnconfinedTestDispatcher: Coroutines run eagerly on the current thread as soon as they are launched. Simpler for tests that don't care about ordering. Good default for most ViewModel tests.

Dispatchers.setMain / resetMain: Must be paired — always reset in @After (or use MainDispatcherRule which does this automatically).

advanceTimeBy: Useful for testing delay(), debounce(), withTimeout() without actually waiting.

Flow & Turbine

📖 Turbine Docs

Turbine makes testing Flow emissions simple. Call flow.test { } and assert items one by one with awaitItem(), awaitComplete(), and awaitError().

Kotlin
📚 Deep Dive

awaitItem(): Suspends until the next emission arrives. Fails the test if nothing is emitted within the timeout (default 1 s).

awaitComplete(): Asserts the flow completed normally. Fails if an error or more items arrive first.

awaitError(): Asserts the flow terminated with an exception.

cancelAndConsumeRemainingEvents(): Cancels collection and returns any unconsumed events — useful when you only care about the first N emissions.

Testing StateFlow: StateFlow always emits the current value on collection. Your first awaitItem() will be the initial state — assert it explicitly before triggering actions.

turbineScope { }: Test multiple flows in parallel without nesting. Each flow gets its own ReceiveTurbine.

ViewModel

📖 Docs

Test ViewModel state by collecting its StateFlow inside runTest. Use MainDispatcherRule to replace Dispatchers.Main, and mock the repository with a fake or Mockito/MockK stub.

Kotlin
📚 Deep Dive

MainDispatcherRule: A JUnit4 TestRule that sets Dispatchers.Main to an UnconfinedTestDispatcher before each test and resets it after. Eliminates the @Before/@After boilerplate.

Fake vs Mock: Prefer hand-written fakes (implement the interface with controllable behavior) over mocks for repositories — they're more readable and don't couple tests to implementation details.

stateIn + WhileSubscribed: In tests, start collecting before calling ViewModel actions. WhileSubscribed(5000) only activates the upstream when there's a subscriber.

SavedStateHandle: Pass a real SavedStateHandle(mapOf("key" to value)) in unit tests — no mocking needed.

Compose UI Tests

📖 Docs

Use createComposeRule() to set Compose content and interact via semantic finders. No Espresso needed — Compose has its own test APIs.

Kotlin
📚 Deep Dive

Finders: onNodeWithText(), onNodeWithTag(), onNodeWithContentDescription(), onNodeWithRole(), onAllNodesWithText().

Actions: performClick(), performTextInput(), performTextClearance(), performScrollTo(), performTouchInput { swipeUp() }.

Assertions: assertIsDisplayed(), assertIsEnabled(), assertIsNotEnabled(), assertTextEquals(), assertContentDescriptionEquals(), assertDoesNotExist().

testTag: Add Modifier.testTag("my_tag") to composables and find with onNodeWithTag("my_tag") — more stable than text-based finders when UI copy changes.

waitUntil: Use composeRule.waitUntil { condition } for async state changes instead of Thread.sleep().

createAndroidComposeRule: Use instead of createComposeRule() when you need access to the real Activity (e.g. to test navigation or system back).

Espresso

📖 Docs

Espresso drives the real Android UI on a device or emulator. Use for View-based screens, navigation flows, or when you need to verify real Activity lifecycle behavior.

Kotlin
📚 Deep Dive

ViewMatchers: withId(R.id.btn), withText("OK"), withContentDescription(), isDisplayed(), isEnabled(), hasSibling(), hasDescendant().

ViewActions: click(), typeText(), replaceText(), clearText(), pressBack(), pressImeActionButton(), scrollTo(), swipeUp().

ViewAssertions: matches(isDisplayed()), matches(withText("...")), doesNotExist().

IdlingResource: Register an IdlingResource to tell Espresso to wait for async work (network, database) before asserting. Without it, assertions run before the UI updates.

Espresso + Compose: In mixed View/Compose screens use both APIs. Access the Compose node tree from Espresso via onView(isRoot()).check { ... } or just use ComposeTestRule alongside Espresso in the same test class.

Interview tip: Espresso synchronizes automatically with the main thread and AsyncTask (legacy). For coroutines/Retrofit you need a custom IdlingResource or OkHttp IdlingResource.
