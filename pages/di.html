Hilt

📖 Docs

Google's recommended DI library. Built on Dagger, but removes most boilerplate with annotations. Compile-time safe.

Kotlin
📚 Deep Dive

Components & Scopes:

  • @Singleton / SingletonComponent — App lifetime
  • @ActivityScoped / ActivityComponent — Activity lifetime
  • @ViewModelScoped / ViewModelComponent — ViewModel lifetime
  • @FragmentScoped / FragmentComponent — Fragment lifetime

@Binds vs @Provides: Use @Binds (abstract) to bind an interface to its implementation — more efficient than @Provides since it generates less code. Use @Provides when you need to call a constructor or third-party code.

@Qualifier: Use when you need to provide two instances of the same type (e.g., two different OkHttpClients).

Dagger

📖 Docs

Compile-time DI via annotation processing. More boilerplate than Hilt but gives full control. Hilt is built on Dagger.

Kotlin

Koin

📖 Docs

Kotlin-native, DSL-based DI. Runtime resolution (not compile-time). Simpler setup, great for Kotlin Multiplatform.

Kotlin

No DI

For simple apps or when learning — manual wiring via singletons, service locator, or an app-level container. Good to know for interviews.

Kotlin
📚 Deep Dive

Service Locator vs DI: Service Locator is an anti-pattern in some views — dependencies are hidden (class pulls them), whereas DI makes them explicit (class receives them). Testability suffers with Service Locator since you must configure the locator in tests.

When no DI is fine: Simple apps with few screens, prototypes, or when you want zero framework overhead. The AppContainer pattern is a clean middle ground.

Interview answer: Mention that even without a DI framework you can still practice the dependency inversion principle by passing dependencies through constructors.
