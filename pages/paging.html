Paging 3

Load large datasets in chunks from network or database. Integrates natively with Compose via collectAsLazyPagingItems().

Setup

📖 Docs

Add the runtime and Compose extension to your version catalog and module gradle file.

TOML / Gradle

PagingSource

📖 Docs

Implement PagingSource<Key, Value> to define how to load a page. Key is typically Int (page number) or String (cursor). Override load() and getRefreshKey().

Kotlin
📚 Deep Dive

LoadResult.Page parameters:

  • data — the list of items for this page
  • prevKey — key for the page before this one (null if first page)
  • nextKey — key for the page after this one (null if last page)

getRefreshKey(): Called when the list is refreshed (e.g. pull-to-refresh). Return the key closest to state.anchorPosition so the user lands on roughly the same item.

Room integration: Room generates a PagingSource automatically when you return PagingSource<Int, Entity> from a @Query DAO method — no manual implementation needed.

invalidate(): Call pagingSource.invalidate() to trigger a full refresh (e.g. after a write operation).

Pager & ViewModel

📖 Docs

Create a Pager in the ViewModel and expose it as a Flow<PagingData<T>>. Always call .cachedIn(viewModelScope) to survive recomposition and config changes.

Kotlin
📚 Deep Dive

PagingConfig parameters:

  • pageSize — items loaded per page. Target 1–3× the visible screen count.
  • prefetchDistance — how many items before the end triggers the next load. Defaults to pageSize.
  • initialLoadSize — size of the first load (defaults to pageSize × 3).
  • enablePlaceholders — show placeholder items for unloaded positions (requires knowing total count).
  • maxSize — drop pages from the opposite end when the list grows past this. Defaults to unlimited.

cachedIn(viewModelScope): Without this, every new collector (recomposition, nav back) triggers a fresh network load. cachedIn keeps the most recent PagingData alive in the scope and replays it to new collectors.

Multiple Pagers: Each Pager instance manages its own page cache. Create a new one when the query changes — don't reuse the same Pager with different params.

Load States

📖 Docs

LazyPagingItems.loadState exposes three independent states: refresh (initial/pull-to-refresh), prepend (loading above), and append (loading below).

Kotlin
📚 Deep Dive

Three LoadState slots:

  • loadState.refresh — triggered by PagingData invalidation or lazyItems.refresh(). Show a full-screen spinner here.
  • loadState.append — triggered when the user scrolls near the end. Show a footer spinner.
  • loadState.prepend — triggered when scrolling near the top (less common). Show a header spinner.

LoadState types: LoadState.Loading, LoadState.NotLoading(endOfPaginationReached: Boolean), LoadState.Error(error: Throwable).

Retry: Call lazyItems.retry() on error. Call lazyItems.refresh() to reload from scratch (e.g. pull-to-refresh).

Interview tip: Distinguish refresh vs append — interviewers often ask which state to observe for a loading spinner vs a footer indicator.

RemoteMediator

📖 Docs

Combines network + local database. Room serves the UI; the mediator fetches from network and writes to Room. Use when you need offline support or a single source of truth.

Kotlin
📚 Deep Dive

MediatorResult:

  • MediatorResult.Success(endOfPaginationReached = false) — loaded OK, more pages exist
  • MediatorResult.Success(endOfPaginationReached = true) — no more pages
  • MediatorResult.Error(throwable) — propagates to loadState.append/refresh.error

LoadType: REFRESH clears the DB and reloads from scratch. APPEND loads the next page. PREPEND loads the previous page (rare).

RemoteKeys table: A common pattern stores the next/prev page cursor for each item in a separate RemoteKeys Room table so the mediator knows where to resume after process death.

Flow direction: Network → Room → PagingSource → UI. The UI never calls the network directly; Room invalidates the PagingSource automatically after each insert.
